## VISUAL AUDIT FINDINGS (2026-05-21 PM)

### 🔴 P0 — `Failed to resolve module specifier "svelte/internal/disclose-version"`

**Scope**: TOATE cele 42 extensii active. Fiecare pagină admin loghează
44+ console errors la load (unul per bundle de extensie). UI-ul extensiilor
nu se randează în Studio — efectiv `marketplace` arată extensii instalate
dar paginile lor nu funcționează.

**Cauza root**: extensiile sunt build-uite cu Svelte ca dependență
EXTERNĂ (`rollupOptions.external: [/^svelte\//]` în vite.config). La
load, bundle-loader-ul Studio fetch-uiește bundle-ul ca text, îl rulează
prin Blob URL → dynamic import. Browser-ul vede `import { ... } from
"svelte/internal/disclose-version"` — specifier bare — și nu poate
rezolva fără import map.

**Studio nu are import map** în [`app.html`](packages/studio/src/app.html).
Memory note menționa că există, dar nu mai e (sau n-a fost niciodată).

**Trei căi de fix** (ordonate de la cel mai pragmatic la cel mai
arhitectural):

#### Cale A — Bundle Svelte în fiecare extensie (~150 KB / extensie)
- Modifică template-ul vite.config.ts pentru extensii: scoate `external`
- Re-publish toate 42 extensii
- Pro: simplu, zero magic; extensiile devin self-contained
- Con: ~6 MB extra încărcat la prima vizită cu toate extensiile active
- Effort: 1-2 zile (refactor template + re-build + re-publish)

#### Cale B — Vite plugin care injectează import-map post-build
- Plugin scanează `dist/_app/immutable/chunks/*.js` după build
- Identifică chunk-urile cu exporturi Svelte (e.g. `discloseVersion`)
- Generează `<script type="importmap">` cu mapping subpath → chunk
- Injectează în `app.html` înainte de `%sveltekit.head%`
- Pro: zero modificare extensii; extensiile rămân slim
- Con: fragil — depinde de stabilitatea exporturilor interne Svelte
- Effort: 2-3 zile (plugin + testare cross-version Svelte)

#### Cale C — Studio expose Svelte runtime la URL stabil
- Adaugă `src/lib/svelte-runtime.ts` care re-exportă tot din `svelte/internal/*`
- Vite garantează că ajunge într-un chunk care e build-uit + servit
- Import-map static în app.html mapează `svelte/internal/*` la acest URL
- Pro: control complet, contract stabil între Studio + extensii
- Con: trebuie să enumerăm explicit toate subpath-urile Svelte folosite
- Effort: 1-2 zile

**Recomandare**: Cale A pentru beta (sigur), Cale C pentru v1.1
(arhitectură corectă).

#### 12 failure-uri categorisate:

**Categoria A — npm peer deps neresolvate (8)**

Pattern: extensia declară un dep în `peerDependencies`, engine-ul rulează
`bun install` în `/opt/zveltio/extensions/`, pachetul există fizic, dar
resolver-ul lui Bun din compiled binary nu îl găsește.

| Extensie | Dep lipsă |
|---|---|
| `operations/traceability` | `qrcode` |
| `compliance/ro/efactura` | `kysely` (import din `.ts`) |
| `storage/cloud` | `@aws-sdk/s3-request-presigner` (transitivă) |
| `communications/mail` | `imapflow` |
| `auth/ldap` | `asn1` (transitivă a `ldapts`) |
| `auth/saml` | `node-saml` |
| `content/media` | `@aws-sdk/config-resolver` (transitivă) |

**Workaround user:** restart engine după install activează majoritatea
(dynamic resolver vs cold-start resolver în Bun se comportă diferit).

**Fix engine necesar:** Bun.plugin shim per extensie pentru peer deps,
similar cu shim-urile existente pentru core (`hono`, `zod`, `kysely`,
`@hono/zod-validator`). Refactor mediu.

**Categoria B — Hono router locked (2)** ✓ **FIX-AT în alpha.92**

`content/pdf-viewer` și `developer/views` aveau `register()` gol/minimal
și failure-ul venea din `app.get(bundleUrl, ...)` pentru Studio bundle,
care nu era prins de catch-ul existent (acoperea doar `register`).
Acum wrapped în același try/catch cu deferral.

**Categoria C — User action (1)**

`geospatial/postgis` cere `CREATE EXTENSION postgis` în psql. Documentat
clar în mesajul de eroare, nu e bug.

**Categoria D — Flaky (2)** (ecommerce/store, finance/banking)

Pică în batch dar trec la retry individual. Probabil ordering — depind
de o extensie anterioară să fi terminat hot-load. Nu critic.

## 3. Caracteristici lipsă (vs competitori: Supabase, Pocketbase, Directus)

### P0 — bloker pentru beta

1. **Storage extension built-in** — Supabase/Pocketbase au file storage out-of-box
   cu signed URLs. Avem `storage/cloud` extension dar e rupt (categoria A).
   Beta-ready cere storage funcțional.

2. **Realtime subscription per-record** — avem realtime per-collection (alpha.90),
   dar nu per-record (`WHERE id = $1`). Supabase oferă asta. Util pentru
   "afișează doar update-urile pentru recordul deschis".

3. **Email magic link UI** — engine suportă (via better-auth) dar Studio
   n-are flow vizibil — doar password.

### P1 — important post-beta

4. **OAuth providers UI** — engine acceptă (better-auth), Studio n-are pagina
   "Connect Google/GitHub/etc" în settings.

5. **Webhooks UI builder** — există route, există catalog (webhooks ext), dar
   payload preview + retry-test interactiv ar reduce friction.

6. **Snapshot/restore granular** — backup-ul existent e full DB. Util:
   restore per-collection sau per-record.

7. **Schema diff vizual** — Studio Schema Branches există dar diff-ul e
   text. Side-by-side cu highlighted columns ar fi mai user-friendly.

### P2 — diferențiator competitiv

8. **AI-assisted schema suggestion** — "tell us your business; we'll create
   the collections". Direct value pentru target audience non-tech.

9. **Marketplace ratings & reviews** — extensiile au manifest, dar zero
   social proof pentru un dezvoltator care evaluează.

10. **Hot reload pe modificare cod extension** — există via `__zveltio_dev_reload`
    dar nu e exposed în Studio. Dev velocity win.

## 4. Implementări care pot fi mai bune

### Extension install (cum sugerai tu)

**Curent:** download tarball → unpack → bun install → dynamic import.
Probleme observate:
- Peer deps transitive (categoria A) nu se rezolvă fiabil
- Hot-load partial-fail invizibil pentru user
- Niciun rollback dacă install reușește dar enable pică

**Varianta superioară propusă:**

```
Phase 1: Verify (no side effects)
  - Download manifest
  - Compute dep tree (npm + zveltio deps)
  - Compute disk + memory budget
  - Show user "this will install X packages, ~Y MB"
  
Phase 2: Stage (sandboxed)
  - Download tarball to /tmp/zveltio-staging/<name>
  - npm install in staging
  - Validate manifest signature (if signed)
  - Test-load in isolated context — verify register() doesn't throw
  
Phase 3: Promote (atomic)  
  - Move staging → /opt/zveltio/extensions/<name>
  - Update zv_extension_registry in one transaction
  - triggerReload
  
Phase 4: Verify post-install
  - Hit a probe endpoint exposed by the extension
  - Mark `health: 'ok'` if responds
  
On any failure: clean up staging, no partial state.
```

**Implementare effort:** 1 săptămână. Înlocuiește 3 din 4 categorii de
failure-uri de mai sus.

### Hot-reload feedback

**Curent:** Server-side log, niciun progress în UI.

**Mai bun:** WebSocket channel `extension-install:<name>` care emit:
- `downloading` (cu progress %)
- `installing-deps`
- `validating`
- `registering`
- `success` / `failed`

Studio afișează un live toast cu spinner + status.

### Studio bundle hot-rebuild

**Curent:** Trigger fire-and-forget la enable/disable, fără feedback.

**Mai bun:** Stream output prin SSE; user vede "Studio rebuild: 70% — compiling Svelte modules".

## 5. Static UI/UX sweep

### A11y patterns lipsă/incomplete

```
$ grep -rln "aria-label\|aria-describedby\|role=" packages/studio/src --include="*.svelte" | wc -l
~120 fișiere folosesc aria-*
```

Coverage acceptabil — majoritatea butoanelor icon-only au `aria-label`.

### Probleme identificate:

1. **Modal-uri fără `aria-modal="true"` consistent** — în câteva locuri da, în altele nu. Standardizare necesară.

2. **Toast-uri fără `role="status"`** — momentan se randează ca div generic; screen reader-ele nu le anunță.

3. **`<button>` fără `type="button"`** — pe pagini cu form-uri, click-ul pe oricare buton submitează formularul. Audit necesar pe ~30 fișiere.

4. **Skip-to-content link** — există în (admin)/+layout.svelte (linia 139), dar nu și în (client) sau (intranet). Inconsistent.
