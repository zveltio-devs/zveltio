## 1. Pass 6 — broad security sweep

### 1.1 Per-extension RBAC gate

**Problem.** 24 of 50 official extensions only checked `auth.api.getSession(...)`
in their route guards. Any authenticated user could read or write the entire
extension — no role gate, no resource segmentation.

**Fix.** New SDK middleware:

- File: [`packages/sdk/src/extension/permission-gate.ts`](../packages/sdk/src/extension/permission-gate.ts)
- Re-export: added to [`packages/sdk/src/extension/index.ts`](../packages/sdk/src/extension/index.ts)
- Signature: `permissionGate(ctx, resource: string, opts?: { actionOverrides?: Record<string, string> })`
- Behaviour:
  - HTTP method → action mapping: GET/HEAD/OPTIONS → `read`, POST → `create`, PATCH/PUT → `update`, DELETE → `delete`. Unknown method → `delete` (most restrictive).
  - Calls `ctx.checkPermission(user.id, resource, action)`. False → 403.
  - God role bypasses (handled inside `checkPermission`).
  - **OPTIONS preflight is short-circuited** — browsers send no credentials on CORS preflight; gating would 401 the preflight and break the actual cross-origin request.
  - `EXTENSION_RBAC=permissive` env knob: gate becomes audit-mode (logs `WOULD DENY ...` but allows through). Default is `strict`.

**Usage pattern in extensions:**

```ts
import { permissionGate } from '@zveltio/sdk/extension';

app.use('*', async (c, next) => {
  const session = await auth.api.getSession({ headers: c.req.raw.headers });
  if (!session) return c.json({ error: 'Unauthorized' }, 401);
  c.set('user', session.user);
  await next();
});
app.use('*', permissionGate(ctx, 'crm'));  // ← place AFTER auth guard
```

**Applied to 24 extensions** including: `crm`, `hr/{employees,leave,payroll,time-tracking}`,
`finance/{accounting,banking,expenses,invoicing,quotes,subscriptions}`,
`operations/{assets,inventory,pos}`, `projects/{helpdesk,management}`,
`content/media`, `ecommerce/store` (admin-only — public storefront stays open),
`integrations/api-connector`, `workflow/checklists`,
`compliance/ro/{documents,efactura,etransport,procurement,saft}`.

**Defaults seeded.** Migration [`077_extension_rbac_defaults.sql`](../packages/engine/src/db/migrations/sql/) (now folded into `001_initial.sql`) grants:
- `employee` role: read on most resources, create on `leave`/`time-tracking`/`helpdesk`/`expenses`.
- `manager` role: read+create+update on operational extensions; read+update on `approvals`/`leave`/`assets`.

The migration adds `CREATE UNIQUE INDEX idx_zvd_permissions_policy_unique ON zvd_permissions (ptype, COALESCE(v0,''), COALESCE(v1,''), COALESCE(v2,''))` first so the `ON CONFLICT (ptype, ...) DO NOTHING` clauses actually deduplicate — without the index, re-running the migration would silently duplicate every policy row.

**IMPORTANT for follow-up work.** Casbin's matcher is
`g(r.sub, p.sub) && (r.obj == p.obj || p.obj == '*') && (r.act == p.act || p.act == '*')`.
Seeded `p` rows alone do nothing — a user must be mapped to the `employee` or `manager` role via a `g` row first (e.g. `INSERT INTO zvd_permissions (ptype, v0, v1, v2) VALUES ('g', '<user-id>', 'employee', NULL)`). The Studio Roles UI exposes this; no users are mapped by default.

### 1.2 Edge functions subprocess sandbox (opt-in)

**Problem.** The default edge-functions runner uses an in-process Bun Worker
with `lockdownGlobals()` (good for admin-authored functions in a trusted
deployment, but not OS-level isolated for arbitrary multi-tenant code).

**Fix.** New runner: [`packages/engine/src/lib/edge-functions/subprocess-runner.ts`](../packages/engine/src/lib/edge-functions/subprocess-runner.ts)

- Spawns a fresh `Bun.spawn` process per invocation.
- IPC: parent writes a single JSON line on the child's stdin; child returns a single JSON line on stdout and exits.
- Minimal env: only `PATH` + `TMPDIR` passed through. `DATABASE_URL`, `BETTER_AUTH_SECRET`, `FIELD_ENCRYPTION_KEY`, etc. are NOT inherited.
- Wall-clock SIGKILL at `timeoutMs + 3s`.
- Same JS lockdown (`lockdownGlobals()`) inside the child as a defence-in-depth layer.

**Wire-up.** [`edge-function-runner.ts`](../packages/engine/src/lib/edge-function-runner.ts) selects mode via `EDGE_SANDBOX_MODE`:
- `worker` (default): in-process Worker. ~1 ms startup.
- `subprocess`: spawn-per-invocation. ~30 ms startup, OS-level isolation.

Env var documented in [`.env.example`](../.env.example).

### 1.3 `reqDb(c, fallback)` helper

**Problem.** Routes that ran outside the request's tenant transaction
(`c.get('tenantTrx')`) would silently return zero rows on FORCE-RLS'd tables
in multi-tenant deployments.

**Fix.** New helper: [`packages/engine/src/lib/route-db.ts`](../packages/engine/src/lib/route-db.ts)

```ts
import { reqDb } from '../lib/route-db.js';

app.get('/things', async (c) => {
  const tdb = reqDb(c, db);  // ← tenant trx if active, else pool
  return c.json(await tdb.selectFrom('zvd_things').selectAll().execute());
});
```

**Applied to:** `documents.ts`, `notifications.ts`, `import.ts`, `export.ts`,
`media.ts`, `edge-functions.ts`, `revisions.ts`, `saved-queries.ts`.
The AI extension defines an inline equivalent in its own routes.

### 1.4 AI embeddings table-name fix (cosmetic, pre-pass-7 leak)

**Problem.** `/ext/ai/embed` and `/ext/ai/search` wrote/read `zv_ai_embeddings`
with columns `field_name`, `content`, JSONB `metadata`. The actual table from
`ai/engine/migrations/002_embeddings.sql` was named `zvd_ai_embeddings` with
`field`, `text_content`, pgvector `embedding`. Every write silently failed (caught and swallowed).

**Fix.** [`ai/engine/routes/ai.ts`](../../zveltio-extensions/ai/engine/routes/ai.ts) now uses
the canonical table; semantic search falls through to pgvector cosine similarity
first, then pg_trgm on `text_content` if no embedder is configured.

This was the visible bug. Pass 7 found a much more serious latent issue with the same table — see §2.1.

## 2. Pass 7 — deep re-audit (critical multi-tenant leaks)

The pass-6 sweep was broad but missed four production-impacting cross-tenant
data leaks. All fixed in `4fb9191` / `77fdb67`.

### 2.1 AI embeddings cross-tenant leak (CRITICAL)

**Problem.** `zvd_ai_embeddings` had NO `tenant_id` column. Both the embed
insert and the search query filtered only on `collection`, so:

- Tenant A `INSERT INTO zvd_ai_embeddings (collection='contacts', record_id=..., text_content='full text of contact')` 
- Tenant B `POST /ext/ai/search { collection: 'contacts' }` → received tenant A's `text_content` (the raw source row's text) in the response.

**Fix.** New migration in the AI extension (now folded into `001_initial.sql`):

```sql
ALTER TABLE zvd_ai_embeddings
  ADD COLUMN IF NOT EXISTS tenant_id UUID;
ALTER TABLE zvd_ai_embeddings
  ALTER COLUMN tenant_id SET DEFAULT NULLIF(current_setting('zveltio.current_tenant', true), '')::uuid;
CREATE INDEX IF NOT EXISTS idx_zvd_ai_embeddings_tenant
  ON zvd_ai_embeddings (tenant_id, collection);
ALTER TABLE zvd_ai_embeddings ENABLE ROW LEVEL SECURITY;
ALTER TABLE zvd_ai_embeddings FORCE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation_ai_embeddings ON zvd_ai_embeddings
  USING (
    NULLIF(current_setting('zveltio.current_tenant', true), '') IS NULL
    OR tenant_id IS NULL
    OR tenant_id::text = current_setting('zveltio.current_tenant', true)
  )
  WITH CHECK (... same condition ...);
```

**Single-tenant deployments still work** — NULL GUC + NULL tenant_id both match.

**Auto-embedding hook fix.** The hook runs on the GLOBAL pool (event bus, not inside the request transaction), so the GUC is unset and the column DEFAULT would tag every embedding with NULL tenant_id → visible to every tenant. The engine plumbs `tenantId` through `RecordCreatedPayload` / `RecordUpdatedPayload` (in `event-bus.ts`) and the AI subscription passes it explicitly to `triggerEmbedding`. Files:

- [`packages/engine/src/lib/event-bus.ts`](../packages/engine/src/lib/event-bus.ts) — payload type
- [`packages/engine/src/routes/data.ts`](../packages/engine/src/routes/data.ts) — `afterWrite` accepts and forwards `tenantId` from `c.get('tenant')?.id`
- [`zveltio-extensions/ai/engine/index.ts`](../../zveltio-extensions/ai/engine/index.ts) — event handler forwards
- [`zveltio-extensions/ai/engine/lib/ai-embed-hook.ts`](../../zveltio-extensions/ai/engine/lib/ai-embed-hook.ts) — `triggerEmbedding(db, collection, recordId, record, tenantId)`

### 2.2 Query cache cross-tenant leak (CRITICAL)

**Problem.** Cache key was `qc:{collection}:{userId+urlHash}`. A user belonging
to multiple tenants who switched via `X-Tenant-Slug` would hit a cached
response from the OTHER tenant on the next request (same userId + same URL =
same key collision).

**Fix.** [`packages/engine/src/lib/query-cache.ts`](../packages/engine/src/lib/query-cache.ts)

- Cache key namespace is now `qc:{tenantId}:{collection}:{hash}`.
- `buildQueryCacheKey(collection, userId, queryString, tenantId)` signature changed.
- `invalidateQueryCache(collection, tenantId)` scopes invalidation to the writing tenant (no cross-tenant churn).

Caller in [`data.ts`](../packages/engine/src/routes/data.ts) passes `(c.get('tenant') as any)?.id ?? null`.

### 2.3 Realtime fan-out (SSE + WS + bus) crossed tenants (CRITICAL)

**Problem.** Three layers, all broken the same way:

1. SSE `broadcastDataEvent` iterated every connection regardless of tenant.
2. WS `broadcastEvent` iterated every connection regardless of tenant.
3. Cross-instance `realtimeBus` envelope carried no tenant id — receiving engines dispatched the message to every same-collection subscriber.

**Fix.**

- [`packages/engine/src/routes/realtime.ts`](../packages/engine/src/routes/realtime.ts) — `StreamSub` records `tenantId` at subscribe time; `broadcastDataEvent(collection, event, record, tenantId)` strict-compares.
- [`packages/engine/src/routes/ws.ts`](../packages/engine/src/routes/ws.ts) — `WSConnection.tenantId` captured at upgrade (from `c.get('tenant')`); `broadcastEvent(collection, event, data, tenantId)` strict-compares.
- [`packages/engine/src/lib/realtime-bus.ts`](../packages/engine/src/lib/realtime-bus.ts) — `RealtimeBusMessage.tenantId` added; `dispatchToWs` forwards it.
- [`packages/engine/src/routes/data.ts`](../packages/engine/src/routes/data.ts) — `afterWrite` forwards `tenantId` to all three.

Strict equality (`(sub.tenantId ?? null) !== (tenantId ?? null)`) is used so single-tenant connections aren't fed multi-tenant traffic and vice versa.

### 2.4 Subprocess runner hardenings

**Problems found in pass 7:**

1. **TOCTOU symlink** — bootstrap file path was `${TMPDIR}/zveltio-edge-runner-${pid}.mjs`. Attacker with `/tmp` write could pre-place a symlink there before engine boot.
2. **PATH hijack** — `spawn(['bun', ...])` resolved `bun` via the child's PATH. Attacker who controlled any earlier PATH entry could replace `bun` and run code inside the engine's user context every invocation.
3. **No code size cap** — a single `POST /api/edge-functions` could pin many MB of engine memory through the transpile + envelope copy.

**Fixes** in [`subprocess-runner.ts`](../packages/engine/src/lib/edge-functions/subprocess-runner.ts):

```ts
// 1. mkdtemp + chmod (defeat symlink swap)
const bootstrapPath = (() => {
  const dir = mkdtempSync(join(tmpdir(), 'zveltio-edge-'));
  const file = join(dir, 'runner.mjs');
  writeFileSync(file, SUBPROCESS_BOOTSTRAP, { encoding: 'utf-8' });
  try { chmodSync(file, 0o600); } catch { /* Windows */ }
  return file;
})();

// 2. Absolute path to parent's own interpreter (no PATH lookup)
const BUN_BIN = process.execPath;
// later: cmd: [BUN_BIN, 'run', bootstrapPath]

// 3. Size cap (1 MiB matches zod limit on POST/PATCH /api/edge-functions)
const MAX_CODE_BYTES = 1024 * 1024;
if (code.length > MAX_CODE_BYTES) {
  return { ok: false, error: `Code exceeds ${MAX_CODE_BYTES} byte limit...` };
}
```

Plus [`edge-functions.ts`](../packages/engine/src/routes/edge-functions.ts) tightened zod schemas: `code: z.string().min(1).max(1024 * 1024)`, `description: z.string().max(2000)`, `display_name: z.string().min(1).max(200)`, `env_vars: z.record(z.string(), z.string().max(8192))`.

### 2.5 Misc hardenings

- **API-key access log** [`api-key-guard.ts`](../packages/engine/src/middleware/api-key-guard.ts) — silent `.catch(() => {})` swallows replaced with `console.warn(...)` so a broken audit trail is visible.
- **DDL manager cleanup** [`ddl-manager.ts`](../packages/engine/src/lib/ddl-manager.ts) — same pattern for m2m junction drops and relation-metadata cleanup.

## 4. Migration squash (alpha-only)

**Context.** Project is pre-1.0; no extension has shipped to production. Per-version migration files were a leftover from incremental development. Consolidating them now is safe; doing it post-1.0 would silently break upgrades for clients whose `zv_schema_versions` table records partial history.

### 4.1 Engine

| | Before | After |
|---|---|---|
| Files in `packages/engine/src/db/migrations/sql/` | 70 (`000_…` → `077_…`) | 1 (`001_initial.sql`, 114 KB) |
| `embedded.ts` imports | 70 | 1 |

The squash preserves `CREATE TABLE zv_schema_versions` (originally `000_schema_versions.sql`) as the first statement — the loader's bootstrap path in [`applyMigration`](../packages/engine/src/db/migrations/index.ts) tolerates the bare `SELECT` failing on first run before the table exists.

`embedded.ts` regenerated via `bun packages/engine/scripts/gen-embedded-migrations.ts` — kept in lock-step with the SQL directory.

### 4.2 Extensions

| | Before | After |
|---|---|---|
| Total `.sql` files across 51 extensions | 119 | 51 (45 squashed `001_initial.sql` + 6 already-single-file kept as-is) |

The 6 extensions that already had a single migration kept their original names (`001_forms.sql`, `001_mail.sql`, etc.) — only renames produce churn.

### 4.3 Squash format

Each consolidated `001_initial.sql` has the shape:

```sql
-- 001_initial.sql
--
-- Consolidated initial schema for the `crm` extension.
-- Source files (applied in this order):
--   • 001_crm_init.sql
--   • 002_crm_organizations.sql
--   • 003_crm_transactions.sql

-- ── from 001_crm_init.sql ──
CREATE TABLE zvd_contacts (...);
...

-- ── from 002_crm_organizations.sql ──
CREATE TABLE zvd_organizations (...);
...

-- DOWN

-- ── DOWN from 003_crm_transactions.sql ──
DROP TABLE IF EXISTS zvd_transactions;

-- ── DOWN from 002_crm_organizations.sql ──
DROP TABLE IF EXISTS zvd_organizations;

-- ── DOWN from 001_crm_init.sql ──
DROP TABLE IF EXISTS zvd_contacts;
```

UP sections in forward creation order; DOWN sections in REVERSE order so rollback unwinds the latest change first. `parseMigrationFile` in [`packages/engine/src/db/migrations/index.ts`](../packages/engine/src/db/migrations/index.ts) splits on the `-- DOWN` marker.

### 4.4 Convention going forward

Future schema changes ship as new files: `002_*.sql`, `003_*.sql`, … alongside the consolidated `001_initial.sql`. Don't edit `001_initial.sql` — it represents the v1.0 baseline.

## 5. Conventions introduced (rules for new code)

### 5.1 Route-level patterns

- **Every extension route module imports `permissionGate` from `@zveltio/sdk/extension`** and uses it after the auth guard. Resource string matches the extension's logical name (e.g. `'crm'`, `'invoices'`).
- **Every engine route handler that touches `zvd_*` tables uses `reqDb(c, db)`** — never the bare `db` parameter — so multi-tenant FORCE-RLS sees the active tenant GUC.
- **Every write that fires `engineEvents.emit('record.created' | 'record.updated' | 'record.deleted')` populates `tenantId`** in the payload (extracted from `c.get('tenant')?.id`). Downstream subscribers (AI auto-embed, future hooks) rely on it.

### 5.2 Tenant scoping checklist for any new feature

Any code path that branches on collection/resource name and persists or broadcasts state must answer all three:

1. **DB:** does the table have `tenant_id` + FORCE RLS + a `tenant_isolation` policy? If yes, queries must run via `reqDb(c, db)` for `SET LOCAL "zveltio.current_tenant"` to apply.
2. **Cache:** does the key namespace include `tenantId`? If using `buildQueryCacheKey`, pass it.
3. **Realtime:** does the broadcast accept a `tenantId` parameter? If using `broadcastEvent` (WS) or `broadcastDataEvent` (SSE), pass it. If publishing to the bus, set `tenantId` in the envelope.

The pass-7 audit traced exactly these three for the AI embeddings path; the same checklist must be applied for any new shared-namespace surface.

### 5.3 Edge functions

- Admin-authored functions in trusted single-tenant deployments → keep `EDGE_SANDBOX_MODE=worker` (default).
- End-user / marketplace / multi-tenant arbitrary code → set `EDGE_SANDBOX_MODE=subprocess`.

### 5.4 Catch-and-swallow audit

Production code may not silently swallow errors on:
- Audit log writes (use `console.warn(...)` so a broken trail is visible).
- DDL cleanup paths (the m2m junction drop pattern in `ddl-manager.ts` is the reference).
- Webhook signing or verification.

Cache miss / stale-data fallback / soft-feature unavailable paths are acceptable swallow sites (the cache helper, the document-indexer, etc.).

## 6. Outstanding gaps (next agent's work)

### 6.1 Systemic multi-tenant gap in extensions other than AI

Only the AI extension has the `tenant_id` + FORCE RLS treatment. The other 50 extensions store user data on `zv_<feature>_*` system-prefix tables with NO `tenant_id` column. In a multi-tenant deployment, an end-user in tenant A can read every other tenant's data in those extensions via the extension's own routes (which filter only by user/owner, not by tenant).

Highest-risk extensions to address first (sensitive data + actively wired into Studio):

| Extension | Tables | Sensitivity |
|---|---|---|
| `compliance/ro/efactura` | `zv_efactura_invoices`, `zv_efactura_storno`, `zv_efactura_daily_stats` | financial PII |
| `crm` | `zvd_contacts`, `zvd_organizations`, `zvd_transactions` | PII |
| `hr/payroll` | salary data | PII + financial |
| `finance/invoicing` | `zvd_invoices` | financial |
| `hr/leave` + `hr/time-tracking` | employee records | PII |

The fix per extension is identical to what pass 7 did for AI: add `tenant_id UUID` with `DEFAULT NULLIF(current_setting('zveltio.current_tenant', true), '')::uuid`, FORCE RLS, a tenant_isolation policy, and ensure routes use the per-request `reqDb(c, db)`. Each is a focused PR; the AI fix is the template.

### 6.2 Casbin role hierarchy

Migration `077` (now folded into engine `001_initial.sql`) seeds `p` rows but not `g` (user→role) rows. Operators must map users to roles via the Studio Roles UI before the gate has any effect. Document this in [SECURITY.md §11.x](SECURITY.md) — currently only mentioned in the migration's own header comment.

### 6.3 Outstanding catch-swallows

A grep for `.catch(() => {})` still returns ~70 occurrences across the engine. Pass 6 + 7 addressed the security-relevant ones (audit logs, ddl cleanup, AI embeddings). The remainder are mostly intentional cache misses or "last_used_at" stat updates. Any new agent doing a swallow-audit should focus on writes that affect external state (Stripe webhook delivery confirmations, push-notification token cleanup, …) and leave the cache/stat paths alone.

### 6.4 Extension typecheck — pre-existing structural debt

`compliance/ro/efactura/engine/routes.ts:186` uses an `as any` cast on `invoice` to bridge a loose `selectAll()` row → strict `InvoiceData` boundary. `compliance/ro/saft/engine/routes.ts:167-168` does the same with `accounts`/`entries`. `finance/accounting/engine/routes.ts:428-429` casts the filtered row arrays. These are silent regressions waiting to happen — the right fix is to type the DB layer per-collection (the SDK's codegen `@zveltio/sdk/codegen` exists exactly for this) and remove the casts. Not security-relevant; would clean up the next typecheck pass.
